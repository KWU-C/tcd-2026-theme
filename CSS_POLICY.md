# CSS運用ポリシー — tcd-2026-theme

> このドキュメントは、CSSの微調整・追加修正時に崩壊を防ぐための運用ルールです。
> 2026年3月の大規模整理（layoutTyp全廃・detail__main_cont正本化）を基準とします。

---

## 1. 基本方針

- **連番命名を再導入しない**。`layoutTypXX` / `worksTypXX` のような番号付きクラスは廃止済み。新規追加禁止
- **意味ベースの命名だけ使う**。クラス名を見て用途がわかることを最低条件とする
- **責務を混在させない**。外枠レイアウト・本文・カードグリッド・導線は、それぞれ独立して管理する
- **場当たり的な上書きより責務の分離を優先する**。既存クラスに重ねて例外を書く前に、責務の見直しを先に行う

---

## 2. 現在の正本ルール

| 役割 | 正本クラス | 備考 |
|---|---|---|
| 記事本文タイポグラフィ | `.detail__main_cont` | フォント・行間・リスト・画像はここで管理 |
| 記事外枠レイアウト | `.article-shell` / `_Title` / `_Inner` | 白カード・幅・影・タイトル領域 |
| Works実績グリッド | `.works-card-grid` / `_Inner` | front-page の実績3列カード |
| 関連カードグリッド | `.content-card-grid` / `_Inner` | single-works / service / branding 下部関連 |
| アーカイブカードグリッド | `.archive-card-grid` / `_Inner` | archive / taxonomy の一覧 |
| フィルターナビ | `.filter-nav` | カテゴリ切替ナビ |
| アクセス2カラム | `.access-two-col` / `_Inner` | 地図+テキスト2列 |
| サービス紹介グリッド | `.service-card-grid` | トップのサービス一覧 |
| コンセプトセクション | `.section-top-concept` | トップのコンセプト文章 |

**これらのクラスの責務・スタイル値を個別に書き換えない。** 調整が必要なときは理由を確認してから行う。

---

## 3. 微調整時の優先順位

1. **既存クラスで解決できるか確認する**（padding / margin の調整など）
2. **既存責務の中で調整する**（セレクタを追加せず値だけ変える）
3. **新クラス追加は最後の手段**。追加前に「本当に再利用されるか」を問う
4. **`!important` は原則禁止**。使う場合は理由をコメントに残す
5. **1ページ専用の応急処置を共通CSSへ混ぜない**。ページ固有対応は `.body-{page}` スコープなどで分離する

---

## 4. 禁止事項

- HTMLにインラインstyleを書かない（`style="..."` 属性の使用禁止）
- ページごとの見た目調整を `.detail__main_cont` など本文コンポーネントへ持ち込まない
- 意味のないクラス名を増やさない（`box2` / `wrap_new` / `tmp_fix` など）
- 既存クラスの責務を変えない（`.filter-nav` をカード用に流用しない、など）
- 同じ役割のクラスを複数作らない（既存クラスで代替できる場合は流用する）
- 一時しのぎの `margin` / `padding` 調整を連鎖させない（応急処置が積み重なる）
- `worksTypXX` / `layoutTypXX` 命名で新しいクラスを追加しない

---

## 5. 新規クラス追加の判断基準

追加前に以下をすべて確認する：

- [ ] 本当に再利用されるか（1ページだけなら追加しない）
- [ ] 既存クラスの責務を壊さないか
- [ ] クラス名だけで用途が説明できるか
- [ ] ページ専用なら、スコープが明確か（`.body-service .xxx` など）
- [ ] 将来削除しやすいか（依存が少ないか）

---

## 6. 命名ルール

- **形式**: `{役割}-{対象}` のケバブケースを基本とする（例: `article-shell` / `archive-card-grid`）
- **サブ要素**: 親クラス名に `_PascalCase` を付ける（例: `article-shell_Inner` / `article-shell_Title`）
- **`_` と `-` の使い分け**: 親子関係には `_`、語の区切りには `-` を使う。この揺れを増やさない
- **BEM**: 厳密BEMは必須ではないが、意味が読めることを優先する
- **禁止パターン**: `tmp` / `test` / `new2` / `fix` / `old` / 連番サフィックス（`card2` など）

---

## 7. 修正前チェックリスト

- [ ] その修正は本文か、外枠か、カードか、導線か（責務を確認する）
- [ ] 既存セレクタと重複・競合しないか
- [ ] 影響範囲は `single` / `archive` / `page` / `front-page` のどこか
- [ ] PC / SP 両方を確認する必要があるか
- [ ] JS依存要素（BxSlider初期化対象、opacity制御など）に触れていないか
- [ ] `.detail__main_cont` / `.article-shell` の正本を変えることになっていないか

---

## 8. 修正後チェックリスト

- [ ] 旧セレクタと競合していないか
- [ ] 似た責務のクラスを増やしていないか
- [ ] hover / SP / padding / 行間が崩れていないか
- [ ] 今回の変更理由を説明できるか
- [ ] 応急処置になっていないか（同じ修正を将来また繰り返す構造になっていないか）

---

## 9. 保留資産の扱い

以下は現役使用中または意図的保留。**無断で変更・削除しない。**

| クラス | 状態 | 扱い方 |
|---|---|---|
| `.worksTyp03` | 現役・単独動作 | 触るときは単独調査の上、別フェーズで対応 |
| `.serviceTyp12` | page.php ページ専用資産 | ページ内完結・干渉なし。現状維持 |
| `.clearfix` | archive-case.php / inc-post.php で現役使用 | 削除しない。将来の整理は別フェーズ |
| `c-card-tile` BEM化 | 将来候補 | 設計判断が必要。現フェーズでは手を付けない |

---

## 10. 運用原則

- **追加より整理を優先する**：新しいクラスを書く前に既存を見直す
- **変更は小さく分ける**：1修正1意図。複数の目的を1回の変更に詰め込まない
- **変更理由を残す**：コメントまたはcommitメッセージで「なぜ変えたか」を記録する
- **次の人が読める状態を維持する**：セレクタを見て役割がわかるコードを書く
- **正本を育てる**：場当たり的な対処より、正本クラスの完成度を上げる選択をする
